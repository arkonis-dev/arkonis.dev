First-class primitives for agents

ARKONIS extends the Kubernetes API with six new resource types.

ArkAgent like Deployment — manages a pool of agent replicas ▼

Manages a pool of agent instances running the same model, system prompt, and MCP tool servers. Handles scheduling, scaling, and semantic health checking.

spec:
  replicas: 5
  model: claude-sonnet-4-20250514
  systemPrompt: |
    You are a research analyst...
  mcpServers:
    - name: web-search
      url: https://search.mcp.internal/sse
ArkService like Service — routes tasks to available agents ▼

Routes incoming tasks to available agent instances. Decouples task producers from the agent pool with configurable load balancing: round-robin, least-busy, or random.

spec:
  selector:
    arkAgent: research-agent
  routing:
    strategy: least-busy
  ports:
    - protocol: A2A
      port: 8080
ArkSettings like ConfigMap — shared prompt config ▼

Reusable prompt fragments and model settings. Reference from multiple ArkAgents to keep configuration in one place and out of application code.

spec:
  temperature: "0.3"
  outputFormat: structured-json
  promptFragments:
    persona: "Expert analyst..."
    outputRules: "Always cite sources."
ArkFlow novel primitive — declarative multi-agent DAG ▼

Chains agents into directed acyclic graphs where outputs feed into inputs. Supports conditionals, loops, timeouts, and structured JSON passing between steps.

spec:
  steps:
    - name: research
      arkAgent: research-agent
    - name: writer
      dependsOn: [research]
      arkAgent: writer-agent
      inputs:
        prompt: "{{ .steps.research.output }}"
ArkMemory like PVC — persistent agent memory ▼

Configures a persistent memory backend for agent instances. Agents remember context across many tasks, not just within a single conversation. Supports Redis and vector stores.

spec:
  backend: vector-store
  vectorStore:
    provider: qdrant
    endpoint: http://qdrant:6333
    collection: research-memories
ArkEvent like CronJob / Ingress — event triggers ▼

Fires flows on a schedule, on inbound webhook calls, or when another flow produces output. One trigger can fan-out to multiple downstream flows in parallel.

spec:
  source:
    type: cron
    cron: "0 9 * * 1-5"
  flows:
    - name: daily-briefing
      arkFlow: briefing-flow

How it works

Everything fits into the Kubernetes model you already know.

1

Declare your agent in YAML

Describe the model, system prompt, MCP tool servers, and replica count. Same format as any other Kubernetes resource.

2

The operator reconciles

ARKONIS watches for changes and ensures the actual state matches your desired state by creating pods, injecting config, and wiring the task queue.

3

Agents pick up tasks and run

Each agent pod pulls tasks from the queue, calls the configured model with its tools, and writes results back. The operator monitors output quality and restarts agents that degrade.

4

ArgoCD and Flux work out of the box

Promoting a new system prompt is a pull request. Your existing GitOps tooling (ArgoCD, Flux, or anything that speaks kubectl) syncs it to the cluster exactly like any other resource change.

The ark local CLI

Full flow execution without a cluster. Same YAML, same agents, same output.

ark run <flow.yaml>

Execute locally

Run any ArkFlow definition directly from your terminal. No operator, no Redis, no cluster required. Perfect for development and CI pipelines.

ark validate <flow.yaml>

Validate before you run

Checks DAG structure for cycles, validates agent references, verifies dependsOn chains, and reports schema errors — before any model call is made.

--provider auto

Auto-detect provider

Pass --provider auto and the CLI infers the right SDK from the model name: claude-* routes to Anthropic, gpt-* / o3-* routes to OpenAI. No config required.

--watch

Stream step output

Print each step's status and a truncated output snippet as it completes. Shows token usage and wall-clock time per step, plus a flow summary at the end.

Semantic health checks

Standard liveness probes check whether a process is running. That's not enough for agents. An agent can be running fine while consistently producing wrong output.

ARKONIS introduces semantic readiness probes that periodically send a validation prompt to the agent and verify the response. Kubernetes stops routing tasks to agents that fail the check. 

livenessProbe:
  type: semantic
  intervalSeconds: 30
GET /healthz Process is up
GET /readyz Agent produces correct output
