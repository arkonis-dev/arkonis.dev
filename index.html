First-class primitives for agents

agent-operator extends the Kubernetes API with four new resource types.

AgentDeployment
like Deployment

Manages a pool of agent instances running the same model, prompt, and tool configuration. Handles scheduling, scaling, and health checking.

spec:
  replicas: 5
  model: claude-sonnet-4-20250514
  systemPrompt: |
    You are a research analyst...
  mcpServers:
    - name: web-search
      url: https://search.mcp.internal/sse
AgentService
like Service

Routes incoming tasks to available agent instances. Decouples task producers from the agent pool with configurable load balancing strategies.

spec:
  selector:
    agentDeployment: research-agent
  routing:
    strategy: least-busy
  ports:
    - protocol: A2A
      port: 8080
AgentConfig
like ConfigMap

Reusable configuration fragments — prompt snippets, model settings, output schemas. Reference from multiple AgentDeployments to keep config DRY.

spec:
  temperature: "0.3"
  outputFormat: structured-json
  promptFragments:
    persona: "Expert analyst..."
    outputRules: "Always cite sources."
AgentPipeline
novel primitive

Chains agents into directed acyclic graphs where outputs feed into inputs. The primitive for orchestrating multi-agent workflows declaratively.

spec:
  steps:
    - name: research
      agentDeployment: research-agent
    - name: writer
      dependsOn: [research]
      agentDeployment: writer-agent

How it works

Everything fits into the Kubernetes model you already know.

1

Declare your agent in YAML

Describe the model, system prompt, MCP tool servers, and replica count. Same format as any other Kubernetes resource.

2

The operator reconciles

agent-operator watches for changes and ensures the actual state matches your desired state — creating pods, injecting config, and wiring the task queue.

3

Agents poll and process

Each agent pod reads tasks from a Redis Streams queue, calls the Anthropic API in a tool-use loop, and writes results back. The operator handles health checks and restarts.

4

GitOps and rollback just work

Promoting a new system prompt is a pull request. Rolling back a bad prompt change is git revert. Audit history is your git log.

Semantic health checks

Standard liveness probes check whether a process is running. That's not enough for agents — an agent can be running fine while consistently producing wrong output.

agent-operator introduces semantic readiness probes that periodically send a validation prompt to the agent and verify the response. Kubernetes stops routing tasks to agents that fail the check.

livenessProbe:
  type: semantic
  intervalSeconds: 30
GET /healthz Process is up
GET /readyz Agent produces correct output
